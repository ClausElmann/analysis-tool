## Gaps Requiring Action

### 🔴 CRITICAL: `create_warning`

**Difference:** L0 accepts Warning ingestion from external systems — validates SourceRef dedup, inserts Warning with Status=New, attaches typed WarningFields and WarningRecipients. GreenAI has no create endpoint, no Warning entity, no ingestion pipeline.

**Impact:** External systems (municipalities, utilities) cannot push alert events into GreenAI. The entire Warnings domain is inoperable.

**Required Action:** Implement CreateWarning slice: validate ProfileId + WarningTypeId scope, insert Warning row, attach fields and recipients

### 🔴 CRITICAL: `warning_template_management`

**Difference:** L0 binds (ProfileId + WarningTypeId) to a MessageTemplate via WarningTemplates — full CRUD. GreenAI has no WarningTemplate entity or management endpoints.

**Impact:** Without template binding, the processing pipeline cannot determine which broadcast template to use for each alert type. Warnings cannot be sent.

**Required Action:** Implement full CRUD for WarningTemplates: GetByProfile, GetByProfileAndType, Insert, Update, Delete

### 🔴 CRITICAL: `warning_recipient_resolution`

**Difference:** L0 resolves phone/email recipients via three strategies: KVHX address lookup, StandardReceiver/Group, and explicit numbers — driven by strategy flags per Warning. GreenAI has no recipient resolution logic.

**Impact:** Even if warnings are created, they cannot be delivered to anyone. Broadcasts cannot be created without resolved recipients.

**Required Action:** Implement recipient resolution service: evaluate ShouldSendToAddress/Owner/IncludedNumber flags, resolve via Lookup domain, enrich WarningMessageDto

### 🔴 CRITICAL: `warning_state_machine`

**Difference:** L0 manages full Warning lifecycle: New(0)→InProgress(4)→FirstRetry(1)→InProgressFirstRetry(5)→SecondRetry(2)→Done(10)/Failed(3), plus Postponed(7/9) for night-time holds. GreenAI has no Warning entity or state transitions.

**Impact:** No retry logic. Failed deliveries are silently lost. No audit trail of processing attempts. Postponement (night-time hold) impossible.

**Required Action:** Implement WarningStatus enum and state transition logic in the processing pipeline

### 🔴 CRITICAL: `warning_processing_pipeline`

**Difference:** L0 runs a batch processing engine: re-queues postponed warnings, loads unprocessed warnings in chunks, resolves recipients, merges fields, creates broadcasts, writes status transitions. GreenAI has no processing pipeline for Warnings.

**Impact:** Warnings are accepted but never processed or delivered. The domain is a dead end without the processing pipeline.

**Required Action:** Implement WarningWorkload processor as a background job: load→resolve→merge→broadcast→status-write cycle

### 🟠 HIGH: `list_warnings_by_profile`

**Difference:** L0 returns historical Warning records filtered by ProfileId + date range, including Status for audit trail. GreenAI has no Warning query endpoint.

**Impact:** Operators cannot view warning history or current processing status. No audit trail available.

**Required Action:** Implement ListWarnings slice filtered by ProfileId + date range

### 🟠 HIGH: `warning_type_management`

**Difference:** L0 exposes WarningTypes scoped to a profile, optionally filtered by ProfileRoleName. GreenAI has no WarningType read endpoint.

**Impact:** UI cannot present available alert categories to operators. Profile-role access restrictions cannot be enforced.

**Required Action:** Implement GetWarningTypes slice — join WarningTypes with ProfileRoleMappings, filter by optional ProfileRoleName

### 🟠 HIGH: `warning_profile_settings`

**Difference:** L0 stores per-profile Warning configuration (night-time postponement windows, default recipient strategy) in WarningProfileSettings. GreenAI has no settings entity or management endpoints.

**Impact:** Postponement logic cannot be configured. Default recipient strategy is undefined. Processing pipeline lacks required configuration context.

**Required Action:** Implement GetWarningProfileSettings + UpdateWarningProfileSettings slices

### 🟠 HIGH: `warning_field_merging`

**Difference:** L0 merges WarningFields (key/value pairs from external source) into the WarningTemplate body at send time. GreenAI has no field merge mechanism for Warnings.

**Impact:** Broadcast messages cannot be personalized with incident-specific data (location, time, severity). Recipients receive generic or incomplete messages.

**Required Action:** Implement WarningField attachment and template merge at broadcast creation time
