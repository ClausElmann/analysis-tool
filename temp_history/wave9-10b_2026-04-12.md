## WAVE 8 — SLICE 1: Create + Submit Broadcast

### Domain Model

```
BroadcastId       : strongly-typed wrapper over int (or Guid — caller-opaque)

BroadcastState    : enum  Draft | Submitted | Targeting | Targeted | Scheduled
                         | Dispatching | Completed | Cancelled

  // Valid transitions:
  //   Draft        → Submitted   (via Submit())
  //   Submitted    → Targeting   (via MarkTargeting())
  //   Targeting    → Targeted    (via MarkTargeted())
  //   Targeted     → Dispatching (Wave 9+)
  //   Dispatching  → Completed   (Wave 9+)
  //   * → Cancelled              (Wave 9+)

BroadcastMessage  : value type
  Text:          string   // required, non-empty
  SenderName:    string?  // optional

RecipientSpec     : value type
  CountryId:         int
  GeographicFilter:  GeographicFilter   // opaque to this slice — just carried
  AddressRestriction: AddressRestriction

Broadcast         : aggregate root
  BroadcastId:       BroadcastId
  ProfileId:         int
  CountryId:         int
  Message:           BroadcastMessage
  RecipientSpec:     RecipientSpec
  Priority:          DeliveryPriority
  State:             BroadcastState
  CreatedAtUtc:      DateTime

  // Methods:
  Submit() → void
    pre:  State == Draft
    post: State = Submitted
    emits: BroadcastReady(BroadcastId)
    throws: InvalidOperationException if State != Draft

  MarkTargeting() → void
    pre:  State == Submitted
    post: State = Targeting
    throws: InvalidOperationException if State != Submitted

  MarkTargeted() → void
    pre:  State == Targeting
    post: State = Targeted
    throws: InvalidOperationException if State != Targeting
```

### Commands + Handlers

```
// Command
CreateBroadcastCommand:
  ProfileId:          int
  CountryId:          int
  Message:            BroadcastMessage
  RecipientSpec:      RecipientSpec
  Priority:           DeliveryPriority
  AddressRestriction: AddressRestriction

// Handler
Handle(CreateBroadcastCommand cmd) → BroadcastId

  Step 1 — ValidatePermission
    permissionSet = IPermissionService.GetPermissionSet(cmd.ProfileId)
    if not permissionSet.Roles.Contains(CanSendBroadcast):
      throw PermissionDeniedException(cmd.ProfileId, CanSendBroadcast)
    trace.Add(PipelineStepResult.Ok("ValidatePermission", inputCount:1, outputCount:1))

  Step 2 — CreateBroadcast
    broadcast = new Broadcast(cmd.ProfileId, cmd.CountryId, cmd.Message,
                               cmd.RecipientSpec, cmd.Priority, cmd.AddressRestriction)
    // state = Draft on construction
    IBroadcastRepository.Save(broadcast)
    trace.Add(PipelineStepResult.Ok("CreateBroadcast", inputCount:1, outputCount:1))

  Step 3 — SubmitBroadcast
    broadcast.Submit()
    // state transitions Draft → Submitted
    // emits BroadcastReady(broadcastId) via domain event on aggregate
    IBroadcastRepository.Save(broadcast)
    trace.Add(PipelineStepResult.Ok("SubmitBroadcast", inputCount:1, outputCount:1))

  return broadcast.BroadcastId

// NOTE: BroadcastReady is a domain event on Broadcast aggregate.
// It is raised inside Submit() and dispatched by the handler after Save.
// It is an EXPLICIT in-process call to the Targeting Engine — NOT an event bus.
```

  Save(broadcast: Broadcast) → void

```

### Commands + Handlers

```
// Command
TargetBroadcastCommand:
  BroadcastId:   BroadcastId
  // RecipientSpec and AddressRestriction read from Broadcast aggregate

// Handler
Handle(TargetBroadcastCommand cmd) → PipelineTrace

  // Called explicitly by the BroadcastReady signal handler — NOT from an event bus

  Step 1 — ReadRecipientSpec
    broadcast = IBroadcastRepository.GetById(cmd.BroadcastId)
    spec = broadcast.RecipientSpec
    // countryId = DK, geographicFilter, addressRestriction
    trace.Add(PipelineStepResult.Ok("ReadRecipientSpec", inputCount:1, outputCount:1))

  Step 1b — MarkBroadcastTargeting
    broadcast.MarkTargeting()    // state: Submitted → Targeting
    IBroadcastRepository.Save(broadcast)
    trace.Add(PipelineStepResult.Ok("MarkBroadcastTargeting", inputCount:1, outputCount:1))

  Step 2 — ResolveAddresses
    result = IAddressService.GetAddressesByGeography(
               countryId=spec.CountryId,
               filter=spec.GeographicFilter,
               restriction=spec.AddressRestriction,
               customerId=broadcast.CustomerId)
    if result is Failed:
      trace.Add(result)   // carries stepName, isRetryable
      return trace        // abort — caller may retry
    addresses = result.Value   // IReadOnlyCollection<AddressRecord>
    trace.Add(PipelineStepResult.Ok("ResolveAddresses", inputCount:1, outputCount:addresses.Count))

  Step 3 — FilterVirtualAddresses
    nonVirtual = addresses.Where(a => a.IsVirtual == false)
    trace.Add(PipelineStepResult.Ok("FilterVirtualAddresses",
               inputCount:addresses.Count, outputCount:nonVirtual.Count))

  Step 4 — ApplyAddressRestriction
    filtered = spec.AddressRestriction == None
               ? nonVirtual
               : nonVirtual.Where(a => criticalAddressIds.Contains(a.Kvhx))
    // For DK path with restriction=None: filtered == nonVirtual (no change)
    trace.Add(PipelineStepResult.Ok("ApplyAddressRestriction",
               inputCount:nonVirtual.Count, outputCount:filtered.Count))

  Step 5 — CreateDeliveryTargets
    targets = filtered.Select(addr => new DeliveryTarget(
                BroadcastId:  cmd.BroadcastId,
                State:        ReadyForDispatch,
                SourceType:   GeographicAddress,
                Kvhx:         addr.Kvhx,
                PhoneCode:    null,   // OBS 2: phone not resolved here
                PhoneNumber:  null,   // OBS 2: phone not resolved here
                Priority:     broadcast.Priority,
                RetryCount:   0))
    IDeliveryTargetRepository.SaveAll(targets)
    trace.Add(PipelineStepResult.Ok("CreateDeliveryTargets",
               inputCount:filtered.Count, outputCount:targets.Count))

  Step 6 — MarkBroadcastTargeted
    broadcast.MarkTargeted()    // state: Targeting → Targeted
    IBroadcastRepository.Save(broadcast)
    trace.Add(PipelineStepResult.Ok("MarkBroadcastTargeted", inputCount:1, outputCount:1))
    // pre-condition enforced: MarkTargeting() MUST have been called (State==Targeting)
    // → non-reproducible execution order is a hard architectural violation

  return trace
```

## WAVE 8 — SLICE 3: Dispatch Dry-Run

### Domain Model (additions to Slice 1+2)

```
ClaimResult  : enum  Ok | AlreadyClaimed | NotFound | Cancelled

DispatchResult : discriminated union
  Sent(gatewayRef: string, sentAtUtc: DateTime)
  Failed(reason: string, isRetryable: bool)
  Skipped(reason: string)

DeliveryLogEntry : value type (append-only)
  DeliveryTargetId:  int
  FromState:         DeliveryState
  ToState:           DeliveryState
  Action:            string      // e.g. "DryRunSkipped"
  TimestampUtc:      DateTime
```

### Commands + Handlers

```
// Command
ClaimAndSendCommand:
  DeliveryTargetId:  int
  DryRun:            bool

// Handler
Handle(ClaimAndSendCommand cmd) → (DispatchResult, PipelineTrace)

  Step 1 — ClaimForDispatch
    claimResult = ITargetingEngine.ClaimForDispatch(cmd.DeliveryTargetId)
    // ITargetingEngine is the ONLY way to enter dispatch lifecycle — LOCK 1

    match claimResult:
      AlreadyClaimed → return (Skipped("AlreadyClaimed"), trace with Skipped step)
      NotFound       → return (Failed("NotFound", isRetryable=false), trace with Failed step)
      Cancelled      → return (Skipped("BroadcastCancelled"), trace with Skipped step)
      Ok             → continue

    // Post-condition: target.State = Claimed
    trace.Add(PipelineStepResult.Ok("ClaimForDispatch", inputCount:1, outputCount:1))

  Step 2 — ExecuteSend
    if cmd.DryRun == true:
      // OBS 1: MUST NOT transition to Sending
      // OBS 1: MUST NOT increment RetryCount
      // OBS 1: MUST NOT trigger DLR logic
      trace.Add(PipelineStepResult.Skipped("ExecuteSend", inputCount:1, outputCount:0, reason:"dryRun=true"))
      dispatchResult = DispatchResult.Skipped("dryRun=true")
    else:
      // NOT implemented in this slice — future wave
      throw NotImplementedException("Live gateway dispatch is Wave 9+")

  Step 3 — LogTransition
    entry = new DeliveryLogEntry(
      DeliveryTargetId: cmd.DeliveryTargetId,
      FromState:        ReadyForDispatch,
      ToState:          Claimed,
      Action:           "DryRunSkipped",
      TimestampUtc:     DateTime.UtcNow)
    IDeliveryLogRepository.Append(entry)
    trace.Add(PipelineStepResult.Ok("LogTransition", inputCount:1, outputCount:1))

  return (dispatchResult, trace)
```

### Service Interface Usage

```
ITargetingEngine:                     // cross-service boundary — LOCK 1 enforcement
  ClaimForDispatch(deliveryTargetId: int) → ClaimResult

IDeliveryLogRepository:               // internal to Dispatch Service
  Append(entry: DeliveryLogEntry) → void
```

### Method Signatures Summary

```
Handle(ClaimAndSendCommand)              → (DispatchResult, PipelineTrace)
ITargetingEngine.ClaimForDispatch(int)  → ClaimResult
IDeliveryLogRepository.Append(entry)    → void
```

  // NOTE: "Delivered" and "PermanentFailure" are NOT valid names in this system.

  // New transitions added in Slice 4:
  //   Claimed → Sending    (gateway accepted submission — BeginSend())
  //   Claimed → Failed     (gateway rejected, retryable — MarkSubmitFailed(isRetryable=true))
  //   Claimed → Abandoned  (gateway rejected, non-retryable — MarkSubmitFailed(isRetryable=false))

GatewayRequest    : value type
  PhoneCode:       string      // from DeliveryTarget
  PhoneNumber:     string      // from DeliveryTarget
  MessageText:     string      // from Broadcast.Message
  SenderName:      string?     // from Profile.SmsSendAs (NOT ForwardingNumber — voice-only)
  IsHighPriority:  bool        // from DeliveryPriority == High

GatewayResponse   : discriminated union
  Accepted(gatewayRef: string)           // submission accepted — gateway will attempt send
  Rejected(reason: string, isRetryable: bool)  // submission refused by gateway

DispatchResult extension (adds to Slice 3 definition — SHARED):
  Submitted(gatewayRef: string, submittedAtUtc: DateTime)
  // Non-terminal: gateway accepted submission. Target state = Sending. DLR pending.
  // DispatchResult.Sent is RESERVED for DLR-confirmed delivery (future DLR slice).
  // Rule: a result type must never assert more than the state machine knows.
  // Sent ≠ Submitted — using Sent here would collapse submission and DLR confirmation.
```

### Commands + Handlers

```
// ClaimAndSendCommand unchanged — DryRun=false now leads to real send path

Handle(ClaimAndSendCommand cmd) → (DispatchResult, PipelineTrace)

  Step 1 — ClaimForDispatch (unchanged from Slice 3)
    claimResult = ITargetingEngine.ClaimForDispatch(cmd.DeliveryTargetId)
    // LOCK 1: sole entry point from Targeting Engine to Dispatch lifecycle
    match claimResult:
      AlreadyClaimed | NotFound | Cancelled → early return (unchanged)
      Ok → continue
    trace.Add(PipelineStepResult.Ok("ClaimForDispatch", inputCount:1, outputCount:1))

  Step 2 — ExecuteSend
    if cmd.DryRun == true:
      // OBS 1: unchanged dry-run behavior from Slice 3
      trace.Add(PipelineStepResult.Skipped("ExecuteSend", inputCount:1, outputCount:0, reason:"dryRun=true"))
      dispatchResult = DispatchResult.Skipped("dryRun=true")
      goto Step 3 — LogTransition

    // === live send path (DryRun=false) ===
    target = IDeliveryTargetRepository.GetById(cmd.DeliveryTargetId)
    // Pre-condition: target.State == Claimed
    // Pre-condition: target.PhoneCode != null  (set by Teledata step or FixedRecipient path)

    gatewayReq = IBroadcastReadModel.BuildGatewayRequest(target)
    // BuildGatewayRequest reads:
    //   Broadcast.Message.Text              → GatewayRequest.MessageText
    //   Broadcast.Message.SenderName        → GatewayRequest.SenderName  (Profile.SmsSendAs — NOT ForwardingNumber)
    //   Broadcast.Priority == High          → GatewayRequest.IsHighPriority
    //   DeliveryTarget.PhoneCode            → GatewayRequest.PhoneCode
    //   DeliveryTarget.PhoneNumber          → GatewayRequest.PhoneNumber
    // MUST NOT read ForwardingNumber — that is a voice-only field (architectural-decisions.md Correction)

    // State remains Claimed. Submit to gateway BEFORE any state transition.
    // Transition occurs AFTER gateway response is known — not before.
    response = ISmsGateway.Send(gatewayReq)

    match response:
      Accepted(gatewayRef):
        // Gateway accepted — atomically transition Claimed → Sending and store gatewayRef
        target.BeginSend(gatewayRef)    // pre: State==Claimed  post: State=Sending, gatewayRef stored
        IDeliveryTargetRepository.Save(target)
        dispatchResult = DispatchResult.Submitted(gatewayRef, DateTime.UtcNow)
        // DispatchResult.Submitted = non-terminal: gateway accepted, state=Sending, DLR pending
        // DispatchResult.Sent is reserved for DLR-confirmed delivery (future DLR slice only)
        trace.Add(PipelineStepResult.Ok("ExecuteSend", inputCount:1, outputCount:1))

      Rejected(reason, isRetryable=false):
        target.MarkSubmitFailed(reason, isRetryable:false) // pre: State==Claimed → Abandoned
        IDeliveryTargetRepository.Save(target)
        dispatchResult = DispatchResult.Failed(reason, isRetryable:false)
        trace.Add(PipelineStepResult.Failed("ExecuteSend", reason, isRetryable:false))

      Rejected(reason, isRetryable=true):
        target.MarkSubmitFailed(reason, isRetryable:true)  // pre: State==Claimed → Failed
        IDeliveryTargetRepository.Save(target)
        dispatchResult = DispatchResult.Failed(reason, isRetryable:true)
        trace.Add(PipelineStepResult.Failed("ExecuteSend", reason, isRetryable:true))

  Step 3 — LogTransition
    entry = new DeliveryLogEntry(
      DeliveryTargetId: cmd.DeliveryTargetId,
      FromState:        Claimed,
      ToState:          target.State,   // Sending | Abandoned | Failed | Claimed (dry-run)
      Action:           [action string]:
        DryRun        → "DryRunSkipped"
        Sending       → "GatewaySubmitted"
        Abandoned     → "GatewayRejectedPermanent"
        Failed        → "GatewayRejectedRetryable"
      TimestampUtc:     DateTime.UtcNow)
    IDeliveryLogRepository.Append(entry)
    trace.Add(PipelineStepResult.Ok("LogTransition", inputCount:1, outputCount:1))

  return (dispatchResult, trace)
```

## WAVE 9 — SLICE 5: Retry Logic (Failed → Claimed, Dispatch-internal)

### CONTRACT CLARIFICATION: Retry ownership vs Lock 1

**Source authorities:** `service-contracts.md` §CONTRACT 2 (LOCK 1), §CONTRACT 3, §SHARED 1.

```
LOCK 1 (verbatim from CONTRACT 2 / Targeting Engine):
  ClaimForDispatch() is the ONLY legal entry point into the Dispatch lifecycle.

SHARED 1 — DeliveryState transition table (relevant entries):
  ReadyForDispatch → Claimed    (ClaimForDispatch() — Targeting Engine → Dispatch Service)
  Failed           → Claimed    (retry — RetryCount++)

These transitions have STRUCTURALLY DISTINCT ownership:

  ReadyForDispatch → Claimed:  Cross-service boundary
    Owner:   Targeting Engine — calls ClaimForDispatch() on its own rows
    Gate:    LOCK 1 — no bypass permitted under any circumstances
    Caller:  Targeting Engine pipeline completion (CONTRACT 2)

  Failed → Claimed:            Dispatch-internal
    Owner:   Dispatch Service owns both Failed and Claimed states (CONTRACT 3)
    Gate:    IncrementAndClaim() — internal method, not gated by ClaimForDispatch
    Caller:  RetryFailed(deliveryTargetId) — a Dispatch Service method (CONTRACT 3)

LOCK 1 governs the cross-service entry path ONLY.
It does not constrain intra-Dispatch state resets.

CONTRACT 3 explicitly lists RetryFailed(deliveryTargetId) as a Dispatch Service method.

Conclusion:
  IncrementAndClaim() (Dispatch-internal, Failed → Claimed, RetryCount++)
  does NOT violate LOCK 1.
  ClaimForDispatch() is NOT called during retry.
```

### Domain Model (additions to Slice 4)

```
// No new state values — Failed and Abandoned already defined in Slice 4

// DeliveryTarget — new field + methods:
  MaxRetryCount:  int   // declared at broadcast creation (default: 3)

  // Dispatch Service owns Failed → Claimed transition for retry (NOT ClaimForDispatch)
  IncrementAndClaim() → void
    pre:  State == Failed  &&  RetryCount < MaxRetryCount
    post: State = Claimed,  RetryCount += 1
    throws: InvalidOperationException if State != Failed
    throws: MaxRetriesExceededException if RetryCount >= MaxRetryCount

  Abandon() → void
    pre:  State == Failed  &&  RetryCount >= MaxRetryCount
    post: State = Abandoned
    throws: InvalidOperationException if State != Failed
```

### Commands + Handlers

```
// Command
RetryDeliveryCommand:
  DeliveryTargetId:  int

// Handler
Handle(RetryDeliveryCommand cmd) → (DispatchResult, PipelineTrace)
// This is a Dispatch Service method. ClaimForDispatch is NOT called here.
// Lock 1 governs entry from ReadyForDispatch. This path starts from Failed.

  Step 1 — LoadAndValidate
    target = IDeliveryTargetRepository.GetById(cmd.DeliveryTargetId)
    if target is null:
      return (Failed("NotFound", isRetryable=false), [LoadAndValidate Failed])
    if target.State != DeliveryState.Failed:
      return (Failed("NotInFailedState", isRetryable=false), trace + [LoadAndValidate Failed])
    trace.Add(PipelineStepResult.Ok("LoadAndValidate", inputCount:1, outputCount:1))

  Step 2 — CheckRetryEligibility
    if target.RetryCount >= target.MaxRetryCount:
      target.Abandon()    // Failed → Abandoned (retry limit reached)
      IDeliveryTargetRepository.Save(target)
      IDeliveryLogRepository.Append(new DeliveryLogEntry(
        DeliveryTargetId: target.Id,
        FromState: Failed, ToState: Abandoned,
        Action: "RetryLimitAbandoned", TimestampUtc: DateTime.UtcNow))
      return (Failed("MaxRetriesExceeded", isRetryable=false),
              trace + [CheckRetryEligibility Failed("MaxRetriesExceeded")])
    trace.Add(PipelineStepResult.Ok("CheckRetryEligibility", inputCount:1, outputCount:1))

  Step 3 — ResetForRetry
    // Dispatch Service owns the Failed → Claimed transition for retry.
    // ClaimForDispatch() is NOT called — that gate is for ReadyForDispatch → Claimed only.
    // This transition is Dispatch-internal, governed by RetryFailed contract (CONTRACT 3).
    target.IncrementAndClaim()    // Failed → Claimed, RetryCount += 1
    IDeliveryTargetRepository.Save(target)
    IDeliveryLogRepository.Append(new DeliveryLogEntry(
      DeliveryTargetId: target.Id,
      FromState: Failed, ToState: Claimed,
      Action: "RetryReset", TimestampUtc: DateTime.UtcNow))
    trace.Add(PipelineStepResult.Ok("ResetForRetry", inputCount:1, outputCount:1))

  Step 4 — DispatchRetry
    // At this point: target.State == Claimed (via IncrementAndClaim)
    // Same send pattern as Slice 4: state remains Claimed until gateway response is known.
    // BeginSend(gatewayRef) called AFTER Accepted — not before. MarkSubmitFailed on Rejected.
    gatewayReq = IBroadcastReadModel.BuildGatewayRequest(target)
    response = ISmsGateway.Send(gatewayReq)

    match response:
      Accepted(gatewayRef):
        target.BeginSend(gatewayRef)   // pre: State==Claimed → Sending (same pattern as Slice 4)
        IDeliveryTargetRepository.Save(target)
        dispatchResult = DispatchResult.Submitted(gatewayRef, DateTime.UtcNow)
        trace.Add(PipelineStepResult.Ok("DispatchRetry", inputCount:1, outputCount:1))

      Rejected(reason, isRetryable):
        target.MarkSubmitFailed(reason, isRetryable)
        IDeliveryTargetRepository.Save(target)
        dispatchResult = DispatchResult.Failed(reason, isRetryable)
        trace.Add(PipelineStepResult.Failed("DispatchRetry", reason, isRetryable))

    IDeliveryLogRepository.Append(new DeliveryLogEntry(
      DeliveryTargetId: target.Id,
      FromState: Claimed, ToState: target.State,
      Action: target.State == Sending ? "RetryGatewaySubmitted" : "RetryGatewayFailed",
      TimestampUtc: DateTime.UtcNow))

  return (dispatchResult, trace)
```

### Section 5: Failure Modes

- `userref` and `Reference` both non-parseable as int

- Outcome: Fatal log, HTTP 200 returned
- SmsLog remains stuck in 1311 indefinitely

**Mode 3 — DLR never arrives (Strex):**
- SmsLog stays at 1311 
- Detected by monitoring after 28–40 hours → Fatal log
- No automatic remediation found in source

**Mode 4 — DLR never arrives (GatewayAPI Bulk):**
- SmsLog stays at 1211
- Detected by monitoring after 12 hours → Fatal log
- No automatic remediation found in source

**Mode 5 — Strex DetailedStatusCode unrecognized:**
- `!Enum.TryParse<StrexDeliveryReportDetailedStatus>(status.DetailedStatusCode, true, out _)` fails

10. **Overdue detection is monitoring/alerting only.** No code found that auto-transitions stuck DLR states. SmsLogs can remain in awaiting states indefinitely if DLR never arrives.

### C. Batch Status Resolution Contract

**Question:** Exact ordering logic when multiple non-final DLR rows exist for one SmsLog.

**Full source trace:**

`MessageService.UpdateStatusOnSmsLogsAsync()` (file: `MessageService.cs` lines 2047–2107):
```csharp
var logsList = _messageRepository.GetNextSmsLogStatusesChunk(300);
// logsList = ALL Imported=0 rows for up to 300 distinct SmsLogIds

var latestStatuses = logsList
    .GroupBy(g => g.SmsLogId)
    .Select(grp => grp
        .OrderBy(s => s.IsFinal ? 3 : s.IsBillable ? 2 : 1)   // priority tier
        .ThenBy(s => s.DateReceivedUtc)                          // within tier: oldest first
        .Last()                                                   // take the LAST = highest priority + latest
    ).ToList();

var latestStatusIds = latestStatuses.Where(l => !l.IsInitial).Select(s => s.Id).ToList();
// IsInitial rows are excluded from the UPDATE (they never write to SmsLogs.StatusCode)

_messageRepository.UpdateSmsLogsStatus(latestStatusIds);
```

`MessageRepository.UpdateSmsLogsStatus()` (file: `MessageRepository.cs` lines 222–233):
```sql
UPDATE SmsLogs
SET StatusCode = st.StatusCode, DateStatusUpdatedUtc = st.DateReceivedUtc
FROM SmsLogs
    INNER JOIN SmsLogStatuses st ON st.SmsLogId = SmsLogs.Id
WHERE st.Id IN @Ids
  AND SmsLogs.StatusCode NOT IN (SELECT Id FROM SmsStatuses WHERE IsFinal = 1)
```

**Priority tier table (from SmsStatuses seed data — verified):**

| Status | IsFinal | IsBillable | IsInitial | Priority tier |
|--------|---------|------------|-----------|--------------|
| 1      | 1       | 1          | 0         | tier 3 (IsFinal) |
| 2      | 0       | 1          | 0         | tier 2 (IsBillable) |
| 3      | 0       | 1          | 0         | tier 2 (IsBillable) |
| 6      | 1       | 1          | 0         | tier 3 |
| 7      | 1       | 1          | 0         | tier 3 |
| 9      | 1       | 1          | 0         | tier 3 |
| 136    | 0       | 1          | 0         | tier 2 |
| 146    | 1       | 1          | 0         | tier 3 |
| 149    | 0       | 1          | 0         | tier 2 |
| 156    | 1       | 1          | 0         | tier 3 |
| 204    | 0       | 0          | 1         | tier 1 (excluded from UPDATE) |
| 206    | 0       | 0          | 0         | tier 1 |
| 10220  | 0       | 0          | 0         | tier 1 |

**Ordering rule (verbatim from `.OrderBy(...).ThenBy(...).Last()`):**

Step 1 — Assign tier: IsFinal=1 → 3, IsBillable=1 (but not IsFinal) → 2, else → 1  
Step 2 — Within same tier: sort by DateReceivedUtc ascending (oldest first)  
Step 3 — `.Last()` = the entry with the HIGHEST tier; within same tier, the LATEST timestamp

**Concrete examples (batch sees all Imported=0 rows at once):**

```
Example 1: Rows [206 @ T1, 1 @ T2]  (Strex: None then Delivered)
  OrderBy: 206 → tier1, 1 → tier3
  ThenBy: within tier3 → only one entry
  .Last(): 1 (tier3 wins)
  → SmsLogs.StatusCode set to 1 ✅

Example 2: Rows [3 @ T1, 1 @ T2]  (GatewayAPI: Accepted then Delivered)
  OrderBy: 3 → tier2, 1 → tier3
  ThenBy: doesn't matter
  .Last(): 1 (tier3 wins)
  → SmsLogs.StatusCode set to 1 ✅

Example 3: Rows [136 @ T1, 156 @ T2]  (Unknown then Undeliverable)
  OrderBy: 136 → tier2, 156 → tier3
  ThenBy: doesn't matter
  .Last(): 156 (tier3 wins)
  → SmsLogs.StatusCode set to 156 ✅

Example 4: Rows [136 @ T1, 136 @ T2]  (duplicate status, different timestamps)
  OrderBy: both → tier2
  ThenBy: T1 first, T2 last
  .Last(): 136 @ T2 (latest within same tier)
  → SmsLogs.StatusCode set to 136 (with DateStatusUpdated = T2) ✅

Example 5: Rows [206 @ T1, 206 @ T2, 1 @ T3]  (Strex: None, None, then Delivered in 3 callbacks)
  OrderBy: 206→tier1, 206→tier1, 1→tier3
  ThenBy: tier3 only has one entry
  .Last(): 1 (tier3 wins regardless of timestamps)
  → SmsLogs.StatusCode set to 1 ✅

Example 6: Rows [204 @ T1] (DuplicateTransaction — IsInitial=1)
  latestStatuses = [204] but IsInitial=true
  latestStatusIds = [] (Where(l => !l.IsInitial) filters it out)
  UpdateSmsLogsStatus called with empty list → no SQL UPDATE
  SmsLogs.StatusCode NOT changed
  → SmsLog retains whatever StatusCode it had ✅

Example 7: Rows [6 @ T1, 1 @ T2]  (Expired then somehow Delivered — both IsFinal)
  OrderBy: 6 → tier3, 1 → tier3
  ThenBy: 6 @ T1 first, 1 @ T2 last
  .Last(): 1 @ T2 (same tier, latest timestamp wins)
  → SmsLogs.StatusCode set to 1
  BUT: UpdateSmsLogsStatus guard checks "NOT IN (IsFinal=1 subquery)"
  If SmsLogs.StatusCode was already set to 6 in a PREVIOUS batch run,
  the guard blocks the UPDATE for 1 as well (6 is IsFinal).
  RESULT DEPENDS ON BATCH TIMING:
    - If both arrive in SAME batch: 1 written (guard: 6 not yet in SmsLogs)
    - If 6 arrives in batch N, 1 arrives in batch N+1: 1 is BLOCKED (6 is IsFinal, guard fires)
```

**Critical contract conclusion:**

```
The batch prioritizes IsFinal over IsBillable over other.
Within the same tier, latest timestamp wins.
IsInitial rows NEVER write to SmsLogs.StatusCode (filtered before UpdateSmsLogsStatus).
The guard on UpdateSmsLogsStatus prevents any overwrite of an already-IsFinal SmsLogs.StatusCode —
  BUT this guard only fires AFTER the previous batch has written the IsFinal status.
  Within a single batch containing multiple IsFinal statuses, the latest-timestamp wins.
```

### D. 10220 Overwrite Risk Classification

**Question:** Can a DLR still arrive after a row reaches 10220? Theoretical or source-confirmed reachable?

**10220 can be reached by two distinct paths:**

**Path 1 — Send failure (all retries exhausted):**
```
Trigger: SharedSmsGatewayTool.GetNextRetryStatus(233) → 10220
  Applied via SmsGatewayStatusWriter.SetStatuses:
    UpdateStatusesOnSmsLogs(233 → 10220) — direct CAS on SmsLogs
    CreateSmsLogStatuses(10220, Imported=true)

Result: SMS was NEVER successfully sent to the gateway.
No gateway reference (Reference field) was transmitted.
GatewayAPI has no record of this message and will NOT send a DLR.

⟹ DLR overwrite risk: THEORETICAL ONLY.
   Physically impossible for GatewayAPI to callback for a message it never received.
   No code path in source creates a GatewayAPI outbound send that then reaches 10220
   via the send-failure chain AND simultaneously sends a message to the gateway.
   (GatewayApiBulkApiWorkloadProcessor: sent=false branch → retry/10220; 
    there is no partial-send scenario visible in this path.)
```

**Path 2 — Strex `Rejected` or `Stopped` DLR → 10220:**
```
Trigger: GatewayFunctions.StrexStatusCode(Rejected) → SmsFailedAtGateway10220
         GatewayFunctions.StrexStatusCode(Stopped) → SmsFailedAtGateway10220
Applied via:
  StrexCallbackAsync → CreateSmsLogStatusAsync → SmsLogStatuses(Imported=0)
  → batch job → UpdateSmsLogsStatus(10220)
  Guard check: SmsLogs.StatusCode NOT IN IsFinal=1 subquery
  10220 NOT in IsFinal=1 set → guard PASSES → 10220 written to SmsLogs.StatusCode

After this, SmsLogs.StatusCode = 10220 (IsFinal=0).

Can a SECOND Strex DLR arrive for the same TransactionId?
  Strex API behavior: If Rejected/Stopped is a terminal state at Strex,
  no further DLRs come. But if Strex retries delivery and updates status,
  a second DLR could arrive.
  Source evidence: StrexDeliveryReportDetailedStatus has 33 values.
  Strex documentation not available in source. Whether Strex sends multiple
  callbacks for the same transactionId is EXTERNAL BEHAVIOR — not determinable from source.

  However: if a second DLR with DetailedStatusCode=Delivered arrives,
  CreateSmsLogStatusAsync inserts a new SmsLogStatuses row (Imported=0),
  batch job processes it:
    - SmsLogs.StatusCode = 10220 (IsFinal=0, not blocked by guard)
    - 10220 → tier1, 1 → tier3
    - 1 wins → SmsLogs updated to StatusCode=1

⟹ Strex path 10220 overwrite: SOURCE-CONFIRMED AS STRUCTURALLY REACHABLE.
   The code path where a 10220 is written via DLR (Rejected/Stopped) and then
   overwritten by a subsequent 1 (Delivered) DLR has ZERO structural barriers.
   All code verified.
```
